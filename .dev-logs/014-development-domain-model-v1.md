# 014 - Mnemo 开发领域层模型 v1

## 背景

在通用层 v2 收敛后，需要先选一个高频、样本充足的领域来做领域层建模。

当前最合适的领域是开发场景，因为：

- 当前 Mnemo 的主要真实使用环境就是 coding agent
- 现有讨论、案例、约束和记忆素材都高度集中在开发场景
- 这能帮助通用层先完成第一次落地校验

因此，本轮先定义开发领域层 v1。

## 核心判断

开发领域层不额外创造一套完全独立于通用层的新记忆体系。

更合理的方式是：

- 通用层定义“记忆的基本类型”
- 开发领域层定义“开发语境下，什么内容会落入这些类型”

也就是说，领域层更像是：

- 通用层在开发场景中的具体映射
- 以及开发场景下的保存边界说明

## 开发领域层映射

### `preference`

开发场景下，主要对应：

- 输出风格偏好
- 方案讨论偏好
- 工程取向偏好
- 协作方式偏好

例子：

- 先讨论方案再写代码
- 偏好简洁、优雅、不过度设计的方案
- 优先给推荐项
- 所有输出和开发日志用中文

### `profile`

开发场景下，主要对应：

- 当前产品/项目背景
- 使用的 agent / 平台环境
- 用户在项目中的长期角色
- 仓库或工作环境的稳定背景

例子：

- Mnemo 是面向 AI coding assistant 的长期记忆项目
- 用户当前同时在规划 metis
- 常用 OpenCode / OpenClaw

### `goal`

开发场景下，主要对应：

- 产品长期方向
- 项目阶段目标
- 中长期的架构/产品目标

例子：

- 把 Mnemo 做成高价值长期上下文系统
- 把触发机制做成 Mnemo 下一阶段第一优先级

### `continuity`

开发场景下，主要对应：

- 尚未收敛完成的架构问题
- 后续必须继续推进的设计线头
- 当前阶段未闭环的项目状态

例子：

- 通用层与领域层模型仍在定义中
- 触发机制设计尚未开始实现
- metis 的产品边界还待继续收敛

### `fact`

开发场景下，主要对应：

- 技术栈
- 仓库路径
- 包名、分支、远程地址
- 固定目录结构
- MCP/tool 支持信息

例子：

- npm 包名是 `@s_s/mnemo`
- 仓库路径是 `/Users/cat/Desktop/packages/mnemo`
- 当前有 `main` 和 `develop` 两个分支

### `decision`

开发场景下，主要对应：

- 设计方案拍板
- 架构取舍结果
- 版本推进顺序确认

例子：

- `scope` 默认值改为 `global`
- `project` 模式使用 `.mnemo/config.json`
- 先做通用层和领域层，再做触发机制

### `rule`

开发场景下，主要对应：

- 工程协作约定
- 提交与日志规范
- 固定工作流
- 输出规则

例子：

- commit 和 push 一起做
- 编号日志可按阶段拆分记录
- roadmap 记录方向，编号日志记录执行阶段

### `experience`

开发场景下，主要对应：

- 被验证过的工程经验
- 可复用的开发教训
- 与特定 agent / 工具 / 工作流相关的长期经验

例子：

- 某类 prompt 提示太抽象时，不足以稳定触发 memory tool
- 某种工作流更能减少遗漏和返工

## 开发领域下不应纳入 Mnemo 的内容

开发场景里有很多信息并不适合进入 Mnemo，即使它们在当下看起来“重要”。

不应纳入的典型内容：

- 普通命令输出
- 一次性调试细节
- 临时任务状态
- 很快会过期的小问题
- 单次报错全文
- 没有长期复用价值的实现过程碎片

原因是这些内容更偏：

- 执行态
- 短期态
- 日志态

而 Mnemo 要保留的是长期上下文，不是工作台完整录像。

## 开发领域层的意义

这层模型的意义不是把开发信息记得更多，而是把开发场景中的长期信息与短期噪音区分开。

它为后续触发机制提供了直接基础：

- 哪些开发场景下应该触发 `memory_save`
- 哪些开发内容虽然重要，但不该进入 Mnemo
- 哪些类型的开发信息更适合将来交给 metis 处理

## 结果

开发领域层 v1 最终确认：

- 开发领域不新增一级记忆类别
- 而是通过通用层 8 类模型完成映射
- 后续触发机制将优先围绕开发场景做第一轮落地

这保证了 Mnemo 仍然是通用产品，而不是只为开发者构建的专用记忆系统。
