# 016 - Mnemo 触发机制 v1

## 背景

在完成以下三个前置阶段后：

- 通用层记忆模型 v2
- 开发领域层模型 v1
- 最小约束机制

现在可以开始定义 Mnemo 的触发机制。

这一步的目标不是立刻改代码，而是先回答：

- 什么时候应该触发 `memory_search`
- 什么时候应该触发 `memory_save`
- 什么时候应该触发 `memory_compress`
- 触发时应该遵守怎样的优先级与约束

## 核心判断

Mnemo 当前最大问题不是缺少工具，而是 agent 不会稳定调用这些工具。

因此，触发机制的核心目标不是“让 agent 记得更多”，而是：

- 让 Mnemo 更稳定地在正确时机被使用
- 让触发动作足够轻，不打断主任务
- 让 Mnemo 作为长期上下文系统自然融入工作流

## 触发原则

### 原则 1：先触发搜索，再触发保存

Mnemo 的默认工作流应该是：

- 先查已有上下文
- 再决定是否产生新记忆

这样可以减少重复保存与重复对齐。

### 原则 2：事件触发优先于抽象判断

与其让 prompt 只写“重要信息时记忆”，不如优先定义明确事件。

事件比抽象原则更容易被 agent 稳定执行。

### 原则 3：触发应尽量贴近自然工作流

Mnemo 不应要求 agent 额外进入一套很重的反思流程。

更合理的方式是：

- 会话开始时自然搜索
- 关键结论后自然保存
- 压缩点前自然检查收敛

## memory_search 触发机制

### 一级触发（应优先发生）

#### 1. 会话开始

当用户发起新一轮任务、问题或讨论时，优先执行 `memory_search`。

目的：

- 先恢复长期上下文
- 避免每次都从零开始

#### 2. 用户引用过去讨论

例如：

- “之前我们说过……”
- “你还记得……”
- “之前定过这个……”

这类场景应优先触发 `memory_search`。

#### 3. 进入长期主题

当话题明显属于一个持续推进中的长期主题时，也应优先搜索。

例如：

- Mnemo 规划
- metis 立项
- 某个持续数天/数周推进的项目主题

### 二级触发（按需发生）

#### 4. 进入熟悉领域但缺背景时

如果当前任务看起来属于此前反复讨论过的领域，但当前上下文不足，也应补一次搜索。

## memory_save 触发机制

### 一级触发（高价值长期上下文）

#### 1. 用户明确表达偏好

例如：

- 输出风格偏好
- 协作偏好
- 决策偏好
- 边界与禁忌

对应类型通常为：

- `preference`

#### 2. 形成明确决策

当某个问题已经拍板，且会影响后续工作，应触发保存。

对应类型通常为：

- `decision`

#### 3. 确认长期规则或工作约定

当讨论中确认了“以后通常怎么做”的默认约定，应保存。

对应类型通常为：

- `rule`

#### 4. 明确长期目标

当用户或项目明确了长期方向，应保存。

对应类型通常为：

- `goal`

#### 5. 产生需要后续继续接上的关键线头

当某个问题未闭环，但明确会继续推进，且下次若不记会丢失连续性，应保存。

对应类型通常为：

- `continuity`

### 二级触发（谨慎保存）

#### 6. 高价值经验被验证成立

只有在经验已经明确可复用、对后续长期协作有价值时，才保存为：

- `experience`

### 不应直接触发保存的内容

- 临时任务状态
- 普通执行输出
- 单次报错全文
- 过程中的猜测
- 尚未确认的判断

## memory_compress 触发机制

### 一级触发

#### 1. 上下文压缩前

当上下文接近压缩点时，应检查是否需要执行 `memory_compress`。

#### 2. 上下文压缩后 / 会话恢复后

恢复后应检查：

- 现有记忆是否需要进一步收敛
- 是否存在大量近似记忆需要合并

#### 3. 长期讨论阶段性结束时

当一段长期话题进入阶段性总结时，也适合触发压缩。

## 触发优先级

### 默认顺序

1. `memory_search`
2. 主任务推进
3. 关键结论后 `memory_save`
4. 阶段结束或压缩点前 `memory_compress`

### 设计原因

- 先搜，避免重复保存
- 先完成主任务，避免记忆动作打断执行
- 在关键节点保存，保证长期价值
- 在阶段结束或压缩点收敛，保持质量

## 开发领域的优先触发点

在开发场景下，以下触发点优先级最高：

- 新会话进入项目讨论时先 `memory_search`
- 用户给出工程规范或协作约定时 `memory_save`
- 架构取舍拍板时 `memory_save`
- 阶段规划调整时 `memory_save`
- 长期话题一轮讨论结束时检查 `memory_compress`

## 与最小约束机制的关系

触发机制不能脱离约束机制独立运行。

也就是说：

- 不是触发到了就一定写
- 而是触发后仍要经过保存阈值与分类判断

所以正确关系是：

- 触发机制负责“什么时候检查”
- 约束机制负责“是否真的写入”

## 结果

Mnemo 触发机制 v1 已确认以下结构：

- `memory_search`：会话开始、引用过去、进入长期主题时优先触发
- `memory_save`：偏好、决策、规则、目标、连续性线头、高价值经验时触发
- `memory_compress`：上下文压缩前后、长期讨论阶段性结束时触发
- 触发顺序遵循：先 search，再 save，最后 compress

这版 v1 先作为后续实现规划的基础，不继续停留在抽象原则层面。
