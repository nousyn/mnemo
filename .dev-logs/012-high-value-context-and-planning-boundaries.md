# 012 - 高价值长期上下文定位与规划边界收敛

## 背景

在继续推进 Mnemo 下个阶段时，最初的思路更偏向于直接增强长期记忆能力，例如检索、压缩、结构化等方向。

但在实际回看使用情况后，发现一个更根本的问题：

- Mnemo 虽然已有完整的 memory tool 能力
- 但真实会话里并没有稳定地产生记忆
- 这说明问题不只是“能力够不够”，而是“Mnemo 到底应该记什么、如何被稳定触发”

因此，这一轮讨论的重点从“直接增强功能”转向“先重定义产品基本盘与建模顺序”。

## 核心收敛结果

### 1. Mnemo 新阶段的基本盘

最终确认的核心表达是：

> Mnemo 不是全量记忆系统，而是高价值长期上下文系统。

这句话的重要意义在于：

- Mnemo 不是要记录所有发生过的内容
- 也不是所有对话都要保存
- 它的目标是保留那些跨会话仍然影响协作、判断、执行和连续性的高价值信息

这为后续所有设计提供了新的约束边界。

### 2. 不再先扩能力，而是先定义记忆模型

确认下个阶段的推进顺序调整为：

1. 先定义通用层记忆模型
2. 再定义领域层记忆模型
3. 再基于模型设计触发机制
4. 最后才进入代码层实现和记忆能力增强

原因是：

- 如果不先定义“Mnemo 应该记什么”
- 那么触发机制一定会过宽、过窄或语义不稳定
- 而后续的检索、压缩、结构化也会缺少统一基准

### 3. Mnemo 必须独立成立

虽然讨论中引出了新项目 `metis` 的方向，以及 `self-improving-agent` 这种 skill 的启发，但最终确认：

- Mnemo 的基本触发与长期记忆能力，必须由自身独立成立
- 不能把“Mnemo 会不会被用起来”这个问题外包给 `metis`
- `metis` 可以增强 Mnemo，但不是 Mnemo 成立的前提条件

这保证了 Mnemo 依然是一个独立产品，而不是另一个系统的附属能力。

## 关于 Skill / MCP 的判断

讨论中也确认了一个重要结论：

- skill 往往比 MCP 更容易被 agent 想起来触发
- 原因不是 skill 能力更强，而是它更接近行为层、更像工作流规则
- 但这并不意味着 Mnemo 应该放弃 MCP 形态

对 Mnemo 的启发是：

- 不应该只把自己当作“一个外部 memory tool”
- 而应该借鉴 skill 的触发机制思路，让 agent 更自然地把 memory_search / memory_save / memory_compress 纳入工作流

## 文档边界重新收敛

这轮还重新确认了文档记录边界：

### `mnemo-roadmap.md`

用途：

- 记录概念层、方向层、长期计划层内容
- 作为后续版本规划和思路提取的长期文档

不应记录：

- 当前完成进度
- 中间状态
- 某一轮讨论推进到了哪里

### 编号日志

用途：

- 记录已经完成的阶段性工作和收敛结果
- 作为灵活留档，不要求必须等整个大阶段结束才写
- 如果任务周期较长，可以拆成多条编号日志分段记录

这次日志本身，就是对本轮“产品重定位 + 规划边界收敛”的一次留档。

## 当前已启动但未完成的内容

在这次收敛后，已经正式开启下一步工作：

- 建立 Mnemo 的通用层记忆模型

目前已提出第一版通用层分类草案，并开始收边界，例如：

- `goal` 与 `open_loop` / `continuity`
- `fact`、`decision`、`rule`

这些内容还处于持续建模阶段，因此暂不在本日志中展开为最终结论，后续会在新的编号日志中继续记录。

## 结果

这次收敛不是代码实现层的推进，而是一次更根本的产品定位修正。

它解决的是：

- Mnemo 到底是什么
- 为什么下一阶段不能先急着改代码
- 为什么要先定义通用层 / 领域层
- 为什么要把 roadmap 与编号日志的边界分开

这为 Mnemo 后续真正进入“高价值长期上下文系统”的建设阶段打下了统一基础。
